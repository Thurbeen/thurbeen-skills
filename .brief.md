## Addendum (added after dispatch)

One observed failure mode during the run that prompted this rewrite: the `admin-xo3` worker opened a PR and then **enabled auto-merge**, causing the PR to merge before the lead could review. This bypassed the user's explicit "verify and validate each thurbox change, notify me when a session needs a review" directive.

**Add to the worker contract (templates/worker-init.md):**

> Workers MUST NOT enable auto-merge on PRs they open. Opening the PR is the terminal action for the worker; the lead (or a human) merges after review. If the `publish` skill offers an auto-merge option, explicitly opt out.

**Add to SKILL.md Phase 5 (Review & notify):**

> Enforcement: the lead NEVER merges a PR it did not review, even if CI is green and auto-merge is enabled. If a worker leaks an auto-merged PR, log a bd note on the bead flagging the violation and proceed with post-hoc review so the user is still informed.
